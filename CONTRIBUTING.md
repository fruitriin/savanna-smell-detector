# savanna-smell-detector コントリビューションガイド

## 基本方針

savanna-smell-detector は **計画駆動（Plan-driven）** のコントリビューションモデルを採用しています。

```
コードではなく計画をレビューする。
筋の良い計画は受け入れ、実装はAIが担保する。
```

---

## コントリビューションの流れ

### 1. Issue / PR の先頭に「きっかけ」を書く

すべてのIssue・PRには、改善のきっかけとなった情報を先頭に含めてください:

```markdown
## きっかけ
- 元となったプロンプト/会話: （概要またはリンク）
- アイデアの出典: （ブログ記事、他プロジェクト、自分の着想など）
- 関連Issue: #XX
```

これはコードの帰属ではなく **アイデアの系譜を追跡する** ためです。

### 2. 計画ファイル（Plan）を含める

実装を伴うPRには、計画ファイルを含めてください:

```
docs/plans/NNNN-タイトル.md
```

計画ファイルには以下を記述:

```markdown
# 計画: タイトル

## 動機
なぜこの変更が必要か

## 設計
どう実装するか

## 影響範囲
どのモジュールに影響するか

## t_wada テスト
この変更は「t_wada の前でも同じこと言えんの？」をパスするか
```

### 3. レビュープロセス

```
計画の筋 → 人間が判断（Accept / Reject / 要議論）
コードの詳細 → CI が担保
```

- **人間がレビューするもの**: 計画の方向性、検出ルールの妥当性、アーキテクチャ上の判断
- **人間がレビューしないもの**: コードのスタイル、実装の詳細、リファクタリングの手法
- **CIが担保するもの**: ビルド、テスト、既存ルールの動作

## 新しいスメルの追加

savanna-smell-detector の最も一般的なコントリビューションは **新しいスメル（検出ルール）の追加** です。

### 判断基準: t_wada が言いそうか？

新しいスメルを提案するとき、以下を自問してください:

> このテストの書き方を見て、t_wada は何と言うだろうか？

t_wada が眉をひそめそうなら、それはスメルです。

### 新スメル追加の手順

1. `SmellType` に variant を追加（`src/core/smell.rs`）
2. `SmellDetector` trait を実装（`src/detectors/`）
3. `default_registry()` に登録（`src/detectors/mod.rs`）
4. テストフィクスチャを追加（`tests/fixtures/`）
5. ビルドが通ることを確認

### 新しい言語パーサーの追加

`LanguageParser` trait を実装し、`src/languages/` に追加してください。

## やらないこと

- コードの行単位でのスタイルレビュー
- 「このメソッド名はこうすべき」等の些末な議論
- 計画なしのコードだけのPR（バグ修正の1行パッチは除く）
